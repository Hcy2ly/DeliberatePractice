3. monorepo概念：https://www.perforce.com/blog/vcs/what-monorepo
   简言之： monorepo（mono 存储库）是一个单独的存储库，用于存储每个项目的所有代码和资产。
   monorepo 将所有内容保存在一个存储库中。multirepo（多个存储库）通常为每个项目有一个存储库。项目越多，存储库就越多。多仓库也称为 polyrepo。
   monorepo适用于：
    能见度-使用单个存储库可让您了解每个项目的代码和资产。这有助于您管理依赖项。
    合作-单个存储库使协作更容易。这是因为每个人都可以访问代码、文件和资产。因此，开发人员可以共享和重用资产。
    速度-使用单个存储库可以帮助您加速开发。例如，您可以进行原子更改（跨多个项目进行更改的一个操作）。

4. 推荐使用pnpm： [https://juejin.cn/post/6932046455733485575#comment]
   【 pnpm也是一个包管理器，相较于npm/yarn它的优势在于？】

    1. 包安装速度极快
    2. 磁盘空间利用非常高效
    3. 安装简单 npm i -g pnpm

    ps：yarn PnP（Plug’n’Play）安装模式：直接去掉 node_modules，将依赖包内容写在磁盘，节省了 node 文件 I/O 的开销，这样也能提升安装速度。（原理解析：https://loveky.github.io/2019/02/11/yarn-pnp/）

   【 pnmp  VS  yarn PnP 】

    1. 安装依赖的速度更快
    2. pnpm 内部使用基于内容寻址的文件系统来存储磁盘上所有的文件，这个文件系统出色的地方在于:
        1. 不会重复安装同一个包，即使用了100次，也只会安装一次，后面再次使用的时候都会直接使用hardlink（硬连接的作用是允许一个文件拥有多个有效路径名。hardlink概念：https://www.cnblogs.com/itech/archive/2009/04/10/1433052.html）
        2. 即使包版本不同也不会重新安装已有的文件，相似的地方不会重新安装，会根据文件引用hardlink，只安装多余的包。
    3. 支持monorepo（同一个存储库），这是一个很大的不同，如在根目录下 pnpm add A -r, 那么所有的 package 中都会被添加 A 这个依赖，当然也支持 --filter字段来对 package 进行过滤。
    4. 安全性高。package.json中为声明的就不安装。之前在使用 npm/yarn 的时候，由于 node_module 的扁平结构，如果 A 依赖 B， B 依赖 C，那么 A 当中是可以直接使用 C 的，但问题是 A 当中并没有声明 C 这个依赖。因此会出现这种非法访问的情况。但 pnpm 脑洞特别大，自创了一套依赖管理方式，很好地解决了这个问题，保证了安全性。
    5. 项目依赖上 没有扁平化处理目录结构，继续嵌套，但是内部用了链接的方式防止包的重装，既没有打乱依赖嵌套的正常结构，节约了扁平化处理的复杂时间，又提升了依赖下载的速度，节约了依赖下载的空间。非常nice。
    6. 安全上巧妙地规避了非法访问依赖的问题，也就是只要一个包未在 package.json 中声明依赖，那么在项目中是无法访问的。只要声明了就可以访问。yarn和npm5.x以后依赖扁平化管理导致依赖性提升从感知上是让你觉得可以跨级访问，但是一个包的随着更新相应的依赖也会更新，新旧之间容易报错。
    7. 使用基本和npm一致，无缝上手。
